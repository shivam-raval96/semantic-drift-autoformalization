Overview

Formal verification is a promising component of scalable oversight for AI-generated artifacts. But when LLMs translate human intent into formal representations, a dangerous failure mode arises: outputs that are formally valid yet semantically unfaithful to the original intent.1 Standard evaluation pipelines miss this — passing formal checks guarantees internal consistency, not faithfulness.

This project studies that failure mode on a clean ground-truth dataset built from Lean-verified equational theories.

A proof checker confirms the formal statement is internally consistent — it cannot know whether that statement is what the human meant.

Worked Example

A concrete VBU case from group-theory equational reasoning. The human asks for a left inverse property; the model returns a statement that type-checks in Lean but quietly means something else. Switch tabs to see a second AI act as an oversight monitor and catch the drift.

Human intent: "Every element has a left inverse" — for all a, there exists b with b · a = e.

Human

Formalize this in Lean for a group (G, ·, e): every element a has a left inverse — some b with b · a = e.

AI · Translator

Here is the formal statement — it type-checks in Lean:

theorem left_inverse (G) [Group G] (a : G) :
  ∃ b : G, a * b = b * a := ⟨a, rfl⟩  -- ✓ compiles
⚠ Valid but unfaithful

Why it's VBU

The checker is happy — but the model wrote a · b = b · a (a commutes with some b), not b · a = e (a left inverse). It's trivially true by taking b = a. Both the operation and the right-hand side drifted. The proof passes; the meaning is wrong.

Oversight setup: a second model re-derives meaning from the formula and compares it to the stated intent — catching what the proof checker can't.

AI · Translator

Proposed translation of "every element has a left inverse":

theorem left_inverse (a : G) :
  ∃ b : G, a * b = b * a := ⟨a, rfl⟩  -- ✓ Lean accepts
✓ Proof checker: PASS

AI · Oversight monitor

Reading the formula back into English: "for all a, there exists b such that a·b = b·a" — i.e. every element commutes with some element. That's trivially satisfied by b = a.

The intent required b · a = e (an inverse yielding identity). The formula has the wrong operand order and no e on the right. Conclusion: the statement is well-formed but does not encode the intent.

✗ Flagged: VBU — semantic mismatch

Suggested faithful form

theorem left_inverse (a : G) :
  ∃ b : G, b * a = e := ...  -- matches intent

Research Questions

Phenomenon. How often do VBU outputs occur, and how does frequency vary with task difficulty? Which failure types are most common?

Prediction. Can these failures be detected with cheap signals — confidence, self-consistency, semantic similarity, hidden-state probes — before or alongside full verification?2

Mechanism. On a constrained subset, where does the model's internal trajectory diverge between faithful and unfaithful outputs? Are matched pairs separable at specific layers?

A cheap pre-filter would matter most where full verification is expensive or unavailable — flagging suspect translations before paying for a proof.

Approach

  1. Construct a benchmark of translation attempts spanning {faithful, not-faithful} × {valid, not-valid}, focused on the VBU class.
  2. Train and compare lightweight detectors using output-space signals and internal representations.
  3. Run focused mechanistic analyses on matched faithful/unfaithful pairs to locate where semantic breakdown becomes visible inside the model.
Why equational theories? Lean-verified equational theories give a clean ground truth: each statement's validity is machine-checkable, so we can isolate faithfulness as the independent variable.

Timeline — MARS program

  • Prework · 2–4 weeks Background readings, failure taxonomy, a toy end-to-end example, and a minimal pipeline.
  • In-person week · July Benchmark v0 (~100s hand-checked / ~1000s generated examples), first detector baselines, error analysis — an MVP by end of week.
  • Remote phase · Aug–Sep · 12–16 weeks Expand the benchmark, compare detector families, run interpretability studies, do cost–performance analysis vs. full verification, and write up.

Expected Outputs

  • A benchmark / dataset of faithful vs. unfaithful formal translations.
  • Baseline methods for cheaply predicting semantically unfaithful outputs.
  • A publication (paper or substantial research post) with a mechanistic analysis of internal signatures of semantic drift — targeted by October.

Success Criteria

Minimum

A usable benchmark, one above-chance detector, a coherent error taxonomy, and a publishable write-up.

Good

A detector usable in practice as a pre-filter, plus evidence on which signal families are most informative.

Stretch

Cheap detection that generalizes across models / datasets, plus a mechanistic finding on where semantic drift is internally visible — or well-characterized negative results.
