A story encoding is a pair of clauses, first an assertion clause and then a question clause.

Clause forms:
- `!{VARS | DEFS | EQ}` means the workplace custom asserts `EQ` always holds after `DEFS`, for all freely chosen starting things in `VARS`.
- `?{VARS | DEFS | EQ}` means the story asks whether `EQ` must always hold after `DEFS`, for all freely chosen starting things in `VARS`.

Syntax:
- `VARS ::= v1,v2,...,vn`
- `DEFS ::= DEF` or `DEF;DEF;...;DEF`
- `DEF ::= r := (x @ y)`
- `EQ ::= x = y`

Names:
- Each `vi` in `VARS` is a freely chosen starting thing.
- Each `r` on the left side of `:=` is a newly named result.
- Each `x` or `y` on the right side of `:=`, and each side of `=`, must be either a starting thing from `VARS` or a result already defined earlier in the same clause.

Meaning:
- `(x @ y)` denotes one performance of the workplace combining action with `x` as the first participant and `y` as the second participant. Order is significant.
- Definitions in `DEFS` are executed from left to right.
- `!` clauses assert universal regularities.
- `?` clauses ask universal questions and do not assert them.

Grammar spec · 3,418 chars 
The notation describes a system with a binary operation and a set of named entities (signals, pigments, etc.).

**Core Concepts:**

*   **Entities (E):** Represent the "things" being combined (e.g., signals, pigments). These are denoted by single uppercase letters (A, B, C, ...).
*   **Operation (Op):** The combining action. This is a binary operation, denoted by `->`. `X -> Y` means "X is fed through Y" or "Y is poured into X". The order matters.
*   **Variables (V):** Named placeholders for entities or results of operations. These are denoted by `v_n` where `n` is an integer.
*   **Equality (Eq):** Denotes that two entities or results are "the same" (e.g., same color, sound alike). Represented by `==`.

**Structure of a Story Representation:**

A story is represented as a tuple: `(Custom, Question)`.

**1. Custom (C):**
   The custom describes an invariant property of the system. It has the following structure:
   `C = (FreeVars, Procedure, Assertion)`

   *   **FreeVars (FV):** A set of entities that can be chosen "any way at all". These are the initial inputs to the custom's procedure.
       `FV = {E_1, E_2, ..., E_k}`
       Example: `{A, B}` for "Take any two signals at all — call the first A and the second B."

   *   **Procedure (P):** A sequence of steps that define how intermediate and final results are obtained. Each step is an assignment.
       `P = [Step_1, Step_2, ..., Step_m]`
       Each `Step` is of the form `v_n = E_x -> E_y` or `v_n = E_x -> v_j` or `v_n = v_i -> E_y` or `v_n = v_i -> v_j`.
       `E_x`, `E_y` must be elements of `FreeVars`. `v_i`, `v_j` must be variables defined in previous steps of the same procedure.
       Example: `[v_1 = B -> B, v_2 = B -> v_1]`

   *   **Assertion (A):** A statement of guaranteed equality between two entities or results.
       `A = E_x == E_y` or `A = E_x == v_n` or `A = v_n == E_y` or `A = v_n == v_m`.
       `E_x`, `E_y` must be elements of `FreeVars`. `v_n`, `v_m` must be variables defined in the `Procedure`.
       Example: `A == v_2`

**2. Question (Q):**
   The question asks whether a certain property *must* hold. It has the following structure:
   `Q = (FreeVars, Procedure, Query)`

   *   **FreeVars (FV):** A set of entities that can be chosen "any way at all" for the question's procedure.
       `FV = {E_1, E_2, ..., E_k}`
       Example: `{A, B}`

   *   **Procedure (P):** A sequence of steps, identical in structure to the custom's procedure.
       `P = [Step_1, Step_2, ..., Step_m]`
       Example: `[v_1 = A -> B]`

   *   **Query (Qy):** A statement of potential equality, identical in structure to the custom's assertion.
       `Qy = E_x == E_y` or `Qy = E_x == v_n` or `Qy = v_n == E_y` or `Qy = v_n == v_m`.
       Example: `A == v_1`

**Naming Conventions:**

*   Initial chosen entities are named `A`, `B`, `C`, etc., in the order they are introduced in the story (e.g., "first whistle" is `A`, "second hum" is `B`).
*   Intermediate results are named `v_1`, `v_2`, `v_3`, etc., in the order they are created within their respective `Procedure` (custom or question).

    *   **Operations (Ops_C):** A sequence of assignments, each on a new line, defining intermediate variables based on input variables or previously defined intermediate variables. The order of operations matters.
    *   **Equalities (Eq_C):** A comma-separated list of pairs of variables, enclosed in square brackets, indicating that these variables always yield the same result. Each pair is separated by `==`. For example, `[a == R2]`.

    *   **Operations (Ops_Q):** A sequence of assignments, each on a new line, defining intermediate variables based on input variables or previously defined intermediate variables. The order of operations matters.

The notation uses a functional style to represent the combining action and a set-like notation for assertions and questions.

**1. Basic Elements:**
*   **Variables:** Represent starting things (e.g., `A`, `B`, `C`, `D`). These are implicitly "any" of their type.
*   **Combining Action:** Represented by a binary operator `->`. `X -> Y` means "do the action to X and Y", where X is the first participant and Y is the second. The result of `X -> Y` is a new, unnamed entity.
*   **Named Results (Batches/Brews/Grafts/Relays):** Represented by `R_n` where `n` is an integer (e.g., `R_1`, `R_2`). These are assigned the result of a combining action.

**2. Expressions:**
*   **Atomic Expression:** A variable (e.g., `A`).
*   **Compound Expression:** `(Expr1 -> Expr2)` where `Expr1` and `Expr2` are either atomic or compound expressions. Parentheses are used for grouping to ensure unambiguous order of operations.
*   **Assignment:** `R_n = Expr` assigns the result of `Expr` to the named result `R_n`.

**3. Custom (Assertion) Structure:**
The custom is represented as a tuple `(S, C, A)`.
*   `S`: A set of variables representing the freely chosen starting things for the custom (e.g., `{A, B, C}`).
*   `C`: A sequence of assignments representing the steps of the custom. Each assignment `R_n = Expr` defines a named intermediate or final result. The expressions in `C` can refer to variables from `S` or previously defined `R_k` (where `k < n`).
*   `A`: A pair of expressions `(Expr_L, Expr_R)` that are asserted to always be the same. `Expr_L` and `Expr_R` can be variables from `S` or named results from `C`.

**4. Question Structure:**
The question is represented as a tuple `(S, Q, P)`.
*   `S`: A set of variables representing the freely chosen starting things for the question (e.g., `{A, B, C, D}`).
*   `Q`: A sequence of assignments representing the steps of the question. Similar to `C` in the custom, expressions in `Q` can refer to variables from `S` or previously defined `R_k`.
*   `P`: A pair of expressions `(Expr_L, Expr_R)` that are asked to always be the same. `Expr_L` and `Expr_R` can be variables from `S` or named results from `Q`.

**5. Full Story Representation:**
A complete story is represented as a pair `(Custom, Question)`.

**Example of how to read `X -> Y`:**
If the action is "pour X into Y", then `X -> Y` means "pour X into Y".
If the action is "graft X onto Y", then `X -> Y` means "graft X onto Y".
If the action is "feed X through Y", then `X -> Y` means "feed X through Y".

**Example of how to read `(Expr_L, Expr_R)` in `A`:**
`Expr_L` and `Expr_R` always come out the same.

**Example of how to read `(Expr_L, Expr_R)` in `P`:**
Must `Expr_L` and `Expr_R` always come out the same?
